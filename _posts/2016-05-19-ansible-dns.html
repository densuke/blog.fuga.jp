--- layout: post title: 'ansible: ローカルDNSサーバーでドメインブロックを' date: '2016-05-19T05:39:00.001+09:00' author: densuke tags: - サーバー - Ansible - Raspberry Pi modified_time: '2016-05-19T05:39:36.241+09:00' blogger_id: tag:blogger.com,1999:blog-1175194959858851005.post-6187018240315201123 blogger_orig_url: http://blog.fuga.jp/2016/05/ansible-dns.html --- Ansibleでpi3の環境を書き換えてサーバー運用してみるという作業をしてみてます。
設定を見て、小ネタを出してみる不定期企画。今日はDNSまわりです。

私のDNS環境は、いわゆるnsd+unboundの分離運用です。このうち、外部に関係するfuga.jpまわりは多少複雑なためnsdでそれなりに書いて、キャッシュとちょっとした書き換えについてはunboundで処理させてます。

ということで、dnsに関するロールを作成し、その中でそれぞれをincludeさせてます。

(roles/dns/tasks/main.yml)
---
- include: cache.yml
- include: content.yml
- include: domainblock.yml

content.ymlはnsdまわりなので今回は無視しておいて、cache.ymlはunbound導入まわりです。

(roles/dns/tasks/cache.yml)
---
- name: キャッシュ用DNSのインストール
  apt:
    name: unbound

- name: unboundの自動起動を設定
  service:
    name: unbound
    enabled: true

- name: local.confの存在を確認
  stat:
    path: "{{ "{{" }} unbound['local'] }}"
  register: ulocal

- name: ローカル設定が無いので、初期状態を作成
  when: ulocal.stat.exists == false
  template:
    src: "{{ "{{" }} unbound['conf'] }}.j2"
    dest: "{{ "{{" }} unbound['local'] }}"
    mode: 0444
    owner: root
    group: root
    validate: "unbound-checkconf %s"
  notify:
    - unboundの再起動

- name: ローカル解決用スタブの追加
  blockinfile:
    dest: "{{ "{{" }} unbound.local }}"
    marker: "# {mark} Ansible Managed Block({{ "{{" }} item.zone }})"
    block: |
      stub-zone:
        name: "{{ "{{" }} item.zone }}."
        stub-addr: {{ "{{" }} ip }}@{{ "{{" }}nsd.port}}
    validate: "unbound-checkconf %s"
  with_items: "{{ "{{" }} nsd.zones }}"
  notify:
    - unboundの再起動

設定を分離して書けるので、/etc/unbound/unbound.conf.d/local.confで内部利用についての設定の雛形をテンプレートから生成し、その後でnsdを用いた内部(ローカル)ドメインの参照をするようにしています。

むしろ今回のキモはdomainblock.ymlです。

---
- name: ブロック定義のやりなおし(環境変数REBUILD_DNSCACHEで判別)
  file:
    path: "{{ "{{" }} unbound.blockconf }}"
    state: absent
  when: lookup('env', 'REBUILD_DNSCACHE') != ""
- name: ブロック定義用ファイルの確認
  stat:
    path: "{{ "{{" }} unbound.blockconf }}"
  register: blockconf

- name: ブロック定義ファイルの先頭部を準備
  copy:
    dest: "{{ "{{" }} unbound.blockconf }}"
    content: |
      server:
    mode: 0444
    owner: root
    group: root
    validate: "unbound-checkconf %s"
  when: blockconf.stat.exists == false
  notify:
    - unboundの再起動

- name: ブロック対象ゾーン定義の追加
  blockinfile:
    dest: "{{ "{{" }} unbound.blockconf }}"
    marker: "# {mark} Ansible Managed Block({{ "{{" }} item }})"
    block: |
      local-zone: "{{ "{{" }} item }}." static
    validate: "unbound-checkconf %s"
  with_items: "{{ "{{" }} dns.blockzones }}"
  notify:
    - unboundの再起動
  tags:
    - maintain

設定ファイル内で、ブロックしたいドメインを羅列しておきます。

- hosts: all
  vars:
    dns:
      blockzones:
        - ad.atown.jp
        - nakanohito.jp
        - doubleclick.net
        - rd.ane.yahoo.co.jp
        - ...以降、追加したくなったら書き込む

これで、ドメインを追加した後で適用させると、追加されてunboundの再起動の後、squidも再起動するように連携させてます。
ただ、この処理のために全タスクを実行させるのは仰々しいので、日常的なメンテナンス項目に対してはmaintainタグをつけるようにして、その中で作業するようにしてます。
パッケージ更新とブロックドメインのチェック程度になるため、日常的に行えます。

なお、ブロックドメインについては、追加もあれば後で消す可能性があるため、実行時に環境変数(REBUILD_DNSCACHE)を設定することで一度設定を削除して入れなおすという方法で対応できるようにしています。
